Recent Changes

2026-04-29

  • Filter state persists in URL: Filters, search text, and active dashboard tile are now mirrored to the browser URL. Clicking a "Linked Files" link in a check 14 cell and pressing Back restores your previous filter (e.g. "Need Review") instead of resetting to "View All."
  • Check 6 (Link Text) — AAA-only rule removed: The "same link text must not point to different URLs" rule was WCAG Level AAA (not the AA target). It also false-positived on the normal CMS pattern of a logo and a breadcrumb both linking home via slightly different URL forms. Descriptive-text, empty-link, and URL-as-link-text rules are still active.
  • Check 14 (Linked Docs) — tightened detector and file-type rule dropped: The doc-link detector was matching the site-wide bullying-report Google Form footer on every page. It now only matches real document URLs (PDF/DOC/XLS, Google Docs/Sheets/Slides, Drive file links). The "must indicate file type in link text" rule was also removed — it is a WebAIM best practice, not a WCAG AA requirement. Pages with real document links show as Needs Review and link to the per-page Linked Files report for file-level review.
  • Check 15 (Videos) — cc_load_policy rule removed: The rule requiring YouTube embeds to include cc_load_policy=1 was not a WCAG requirement; WCAG 1.2.2 only requires captions to exist. The YouTube CC button satisfies this. Caption quality review is still flagged as manual.
  • Check 12 (Embeds) — iframe name check expanded: The iframe accessible-name check now accepts aria-label and aria-labelledby in addition to title, matching axe-core behavior. Google Maps embeds (which use aria-label but no title) no longer false-positive.
  • Check 4 (Focus Indicator) — bogus outline counter removed: The check was counting elements with outline: none in their unfocused state — which only measures whether the page uses modern CSS (reset outline by default, restore at :focus-visible), not whether focus is actually visible. Replaced with a plain manual-review marker. Focus indicator review is now correctly left to keyboard testing.
  • Check 2 (Reading Order) — CSS order detector removed: The check was reporting every element with a non-zero CSS order or flex-direction: row-reverse. WCAG 1.3.2 is about DOM order, not visual CSS order; Apptegy's Vue template uses flex order for legitimate responsive layout. Replaced with a plain manual-review marker.
  • Check 5 (Alt Text) — over-strict SVG rule removed: The custom rule flagged any <svg> without an accessible name. WCAG 1.1.1 only requires alt text on SVGs that convey information — decorative SVGs are exempt. axe-core's svg-img-alt rule already handles the correct case (svg[role="img"] only).
  • Review UI — school URLs display as <site>.eusd.org: Page URLs now show in the familiar subdomain form (e.g., glenview.eusd.org/…) rather than the internal alias form (www.eusd.org/o/glenview/…). Database is unchanged; purely cosmetic.

What Is This Tool?

This tool audits all EUSD district and school webpages for ADA/WCAG 2.1 Level AA compliance as required by the DOJ web accessibility rule. It runs automated checks on every page, flags issues that need human review, and provides a browser-based interface for working through the review queue. Results are stored in a local database and can be exported to Excel for reporting.

What This Tool Does

The audit runs in four phases. Each phase feeds into the next:

PhaseCommandWhat It Does
1. Import npm run import-linkcheck
npm run import		 Loads EUSD pages into the local database. import-linkcheck (preferred) imports ~1,400 pages from the eusd-linkcheck crawler CSV. import loads pages from the source Excel file. Both are safe to re-run — existing rows are never overwritten.
2. Audit npm run audit Opens each page in a headless browser, runs 15 automated accessibility checks, and records results. Flags anything that needs manual review.
3. Review npm run review-ui Launches a browser-based spreadsheet for reviewing audit results. Click to mark checks pass/fail, add notes. All changes save automatically.
4. Export / Report npm run export
npm run dashboard		 Exports results to Excel or CSV. Generates an HTML summary dashboard.
Note: All EUSD sites run Apptegy CMS (a Vue.js SPA). The tool uses a real browser (Playwright/Chromium) to load pages fully before scanning — plain HTTP fetching would miss dynamically rendered content.

Step-by-Step

1 Import pages from the Excel source file

npm run import

This reads resources/EUSD.org Full Audit.xlsx and populates the database with all pages from 10 school sites plus the district site. Run this once during initial setup, and re-run whenever the spreadsheet is updated with new pages.

2 Run the automated audit

# Audit all pages (runs for a while — use --limit to batch)
npm run audit

# Recommended: audit in batches of 50
npm run audit -- --limit 50

# Audit a single page for testing
npm run audit -- --url https://www.eusd.org/

# Audit only one school site
npm run audit -- --site eusd --limit 50

# Skip pages already audited
npm run audit -- --new

The audit opens each page in a headless Chrome browser, expands Apptegy accordions, injects axe-core, and runs all 15 checks. As it runs, it prints a line per page showing which checks passed, failed, or need review. Results are saved to the database automatically.

Expected output looks like:

Auditing: EUSD Home (https://www.eusd.org/)
  ✓ Skip Links        ✓ Alt-Text          ✓ Color Contrast
  ✓ Link Text         ✗ Heading Structure  ? Visual Focus
  ...
  Saved: 3 pass, 1 fail, 2 review

Audit complete: 50 pages | 41 pass | 6 fail | 3 review | 0 error

3 Review results in the web UI

npm run review-ui

This launches a local web server and opens http://localhost:3000 in your browser. You'll see a spreadsheet-style grid with every page and its 15 checks.

  • Yellow cell (?): Unreviewed — needs a decision
  • Green cell (P): Passed
  • Red cell (F): Failed — needs remediation
  • Click a status cell to cycle through: ? → P → F → ?
  • Click a notes field to type reviewer comments; saves on blur
  • "Pass All" button on a page row marks all unreviewed (?) checks as Pass — it never overwrites checks you've already set

Use the filters at the top to focus your work:

  • Status filter: show only Unreviewed, only Fail, etc.
  • Site filter: focus on one school at a time
  • Check filter: work through one check type across all pages
  • Search: find a specific page by name or URL
Tip: Filter state is saved in the browser URL. If you click a Linked Files link in a check 14 cell to review that page's documents and then press Back, your previous filter (e.g., "Need Review") is automatically restored.
Note: The review UI starts automatically when you log in (via a scheduled task). If it's already running, just open http://localhost:3000 in your browser.

4 Export results

# Export to CSV
npm run export

# Export to Excel (3 sheets: results, summary, manual queue)
npm run export -- --format xlsx

Output files appear in ~/projects/eusd/doj-webrule-audit/output/exports/ with today's date in the filename.

5 Generate the HTML dashboard

npm run dashboard

Creates ~/projects/eusd/doj-webrule-audit/output/dashboard.html. Open it in a browser to see overall progress, by-check breakdown, by-site stats, and daily trend history. The team-facing version is published to GitHub Pages automatically.

The 15 Audit Checks

#Check NameWhat It TestsAutomation
1 KB ACCESS Keyboard accessibility of interactive widgets: click handlers missing keyboard roles, misused tabindex Partial — menus, modals, and dropdowns require manual keyboard testing
2 READING ORDER DOM order matches visual order so screen readers encounter content in a logical sequence Manual — screen reader verification required; CSS property values alone cannot determine this
3 SKIP LINKS First focusable element is a working "skip to main content" link pointing to a valid page ID Full auto
4 VISUAL FOCUS INDICATOR Focusable elements have a visible outline when keyboard focused (WCAG 2.4.7) Manual — requires tabbing through the page; computed styles alone cannot determine focus visibility
5 ALT-TEXT/LABELS Images have meaningful alt text (not filenames, not too long); SVGs have accessible names Full auto
6 LINK TEXT WELL NAMED Links use descriptive text (not "click here", "more", or a raw URL); empty links are flagged Full auto
7 COLOR ALONE Color is not the only visual means of conveying information (e.g., charts must have labels) Partial — visual inspection of charts and status indicators required
8 COLOR CONTRAST Text/background contrast meets 4.5:1 (normal text) or 3:1 (large text) ratio per WCAG AA Full auto (axe-core measures actual pixel colors)
9 TABLES Data tables have <th> headers with scope, a caption or accessible label, and correct semantics Full auto
10 BUTTONS/FORM CONTROLS All inputs and buttons have labels via <label>, aria-label, or visible text; no placeholder-only inputs Full auto
11 HEADING STRUCTURE One <h1> per page; headings in sequential order with no skips; no empty headings Full auto
12 EMBEDDED VIDEOS/CAROUSELS Iframes have an accessible name via title, aria-label, or aria-labelledby; carousels have keyboard controls and pause buttons; no autoplay video Partial — keyboard navigation and pause functionality must be tested manually
13 MAGNIFICATION Viewport meta allows zooming; page is readable and functional at 200–400% magnification Manual — requires zoom testing at different levels
14 LINKED DOCS/PDFS Pages with links to PDFs, Office docs, or Google Docs/Sheets/Slides are flagged for file-level review. Use the Linked Files report (link in the check cell) to review each linked document individually. Partial — document accessibility (tags, reading order, alt text inside the file) requires opening each file
15 VIDEOS HTML5 videos have caption tracks; YouTube embeds must have captions available (WCAG 1.2.2 — captions must exist, not necessarily auto-enabled) Partial — caption quality and accuracy require human verification

Understanding the Output

Review UI (http://localhost:3000)

The main working interface. Shows all pages as rows in a grid with 15 check columns. Status cells are color-coded: yellow (unreviewed), green (pass), red (fail). Notes cells accept free text. All edits auto-save to the database.

Linked Files Report (http://localhost:3000/files.html)

A secondary view listing every linked document (PDF, DOCX, Google Doc, etc.) found across all pages. Accessible via the Linked Files → link in the main header or by clicking the file count in a check 14 cell. For each file you can mark it pass/fail and add notes — this is where per-document accessibility review happens (tagged PDFs, reading order within the file, alt text on images inside the document).

HTML Dashboard (output/dashboard.html)

A summary report showing overall progress, pass rates by check type, results by school site, top critical failures, and daily progress over the last two weeks. Regenerate it any time with npm run dashboard. The team-facing version is published automatically to GitHub Pages.

CSV Export (output/exports/audit-export-YYYY-MM-DD.csv)

One row per page per check (page × 15 checks). Includes site, page name, URL, check name, automated result, manual override, notes, and audit date. Pinned to the latest completed audit run — partial or in-progress runs are excluded.

Excel Export (output/exports/audit-export-YYYY-MM-DD.xlsx)

Three sheets: Audit Results (same as CSV), Summary (one row per page with pass/fail/review counts), and Manual Queue (all pending reviews with priority and reviewer notes).

Troubleshooting

Error / SymptomCauseFix
No completed audit run found Pages imported but no audit has been run yet, or all runs are still in progress Run npm run audit -- --limit 10 to complete a batch
Page load timeout during audit Page took longer than the configured timeout (default 30s) Try PLAYWRIGHT_TIMEOUT=60000 npm run audit, or test the specific URL with npm run audit -- --url <url>
Port 3000 already in use The review UI is already running (possibly from the autostart scheduled task), or another app is on that port Just open http://localhost:3000 — it's already running. Or use REVIEW_PORT=3001 npm run review-ui for a second instance
Cannot find module 'xlsx' or similar Node modules not installed Run npm install in the project folder
SQLITE_BUSY / database locked Multiple processes accessing the database at the same time Close other instances (review-ui, other audit runs) and retry
Review UI doesn't auto-open NO_OPEN=1 is set in environment (used by the scheduled autostart task) Manually navigate to http://localhost:3000
Excel import skips a sheet Expected sheet name is missing from the Excel file Verify the Excel file matches the expected structure and re-export from source if needed
Audit result counts don't match the dashboard Dashboard was generated before the latest audit completed Run npm run dashboard to regenerate with current data
A check you set was overwritten Should not happen — the system never overwrites manually set results Check the notes field for context; contact the administrator if results appear corrupted

Quick Reference Card

Daily Workflow

  1. Open the review UI: navigate to http://localhost:3000 (starts automatically at login)
  2. Run the morning audit batch: npm run audit -- --limit 50
  3. Switch to the review UI, filter to Unreviewed, and work through new results
  4. Click status cells to mark pass/fail; add notes where needed
  5. Use Pass All for pages that look fully accessible
  6. At end of day, regenerate the dashboard: npm run dashboard

Weekly Maintenance

  • Find new pages: npm run discover -- --site eusd --diff
  • Check for duplicate pages: npm run dupes
  • Audit only new/unaudited pages: npm run audit -- --new
  • Export for reporting: npm run export -- --format xlsx
