What This Tool Does

The EUSD ADA Toolkit helps you make web projects accessible to all users, meeting WCAG 2.1 Level AA standards. It provides two tools that work at different stages of a project:

ToolWhat It DoesWhen to Use
Build-time skill Enforces WCAG 2.1 AA rules automatically while Claude Code writes or edits HTML, CSS, and JavaScript. Also includes an on-demand audit command. Any project you're building with Claude Code that has a web interface
Formal audit report Audits any web page and produces a Word document report with every issue found, its severity, and a specific Claude Code prompt to fix it. Auditing live sites, third-party pages, or producing formal documentation
Note: EUSD is legally required to make its websites accessible under ADA Title II and Section 508. These tools help you comply from the start rather than fixing issues after the fact.

Prerequisites

One-Time Setup (already done on your machine)

  1. Claude Code installed and working
  2. Toolkit repo cloned to C:\Users\mberning\projects\eusd\eusd-ada-toolkit

If Setting Up on a New Machine

# 1. Clone the toolkit repo
git clone https://github.com/Escondido-Union-School-District/eusd-ada-toolkit.git

# 2. That's it — the toolkit is just files you copy into your projects

Step-by-Step: Installing the Build-Time Skill

Do this once per project that has a web interface.

1 Create the skill directory in your project

mkdir -p .claude\skills\ada-wcag

2 Copy the skill file from the toolkit repo

copy C:\Users\mberning\projects\eusd\eusd-ada-toolkit\skill\.claude\skills\ada-wcag\SKILL.md .claude\skills\ada-wcag\SKILL.md

3 Add the CLAUDE.md snippet

Open C:\Users\mberning\projects\eusd\eusd-ada-toolkit\skill\CLAUDE-snippet.md and copy everything below the first line into your project's CLAUDE.md. If you don't have a CLAUDE.md yet, create one at your project root.

4 Add .wcag-progress.json to your .gitignore

The audit skill tracks progress in a .wcag-progress.json file. Add it to your project's .gitignore so it doesn't get committed:

# In your project's .gitignore, add this line:
.wcag-progress.json
Warning: This step is easy to forget. The toolkit's own .gitignore has this entry, but it does not carry over when you install the skill into a new project.

5 Verify it's working

Start a Claude Code session in your project and ask it to write or edit some HTML. It should automatically follow accessibility rules (proper heading structure, alt attributes, contrast-safe colors, etc.) without you having to ask.

Step-by-Step: Running the Accessibility Audit

Run this before launching any project with a web interface, or anytime you want a thorough accessibility check.

1 Open your project in Claude Code

Navigate to your project folder and start a Claude Code session.

2 Tell Claude Code to run the audit

run accessibility audit

You can also say check accessibility, wcag audit, or a11y check.

3 Review the results

Claude Code will scan all HTML files (or source components for SPAs), audit each one against the full WCAG 2.1 AA criteria (19 categories covering structure, contrast, keyboard, forms, media, motion, reflow, and more), and present issues sorted by severity:

  • Critical — Fails a Level A criterion or has site-wide impact. Blocks access.
  • Serious — Fails a Level AA criterion. Significant barrier.
  • Moderate — Best practice issue or limited scope. Causes friction.

4 Fix issues interactively

Type fix to start with the first issue, or fix 5 to jump to a specific one. Claude Code shows each problem, explains the impact, and proposes a fix. Type y to apply, s to skip, or q to quit.

Progress is saved in .wcag-progress.json, so you can close the session and resume later.

Step-by-Step: Generating a Formal Audit Report

Use this for live sites, third-party pages, or when you need a formal Word document for records.

1 Open the audit project in Claude Code

cd C:\Users\mberning\projects\eusd\eusd-ada-toolkit\audit-project
claude

2 Follow the prompts

Claude Code will ask you to paste in the page's HTML, CSS, and JavaScript. It walks you through how to grab each file from your browser.

3 Review the Word document

Claude Code generates a report with every issue found, organized by category, with severity ratings and specific fix prompts.

Understanding the Output

Build-time skill

When the skill is installed, Claude Code silently follows WCAG rules as it writes code. You won't see separate output unless it flags an issue during coding.

Accessibility audit

The audit prints a summary to the terminal showing how many issues were found at each severity level, then enters an interactive fix mode. Progress is tracked in .wcag-progress.json.

Formal audit report

Produces a Word document (.docx) with:

  • Every accessibility issue found, organized by WCAG category
  • WCAG criterion reference for each issue
  • Severity rating (Critical, Serious, Moderate)
  • A specific Claude Code prompt for fixing each issue

Troubleshooting

ErrorCauseFix
Claude Code doesn't follow accessibility rules during coding Skill not installed or CLAUDE.md snippet missing Verify .claude\skills\ada-wcag\SKILL.md exists in your project and the snippet is in your CLAUDE.md
"run accessibility audit" does nothing special SKILL.md not in the right location Check that the file is at .claude\skills\ada-wcag\SKILL.md (not a different folder name)
.wcag-progress.json showing up in git status Not added to .gitignore Add .wcag-progress.json to your project's .gitignore
Audit says "no HTML files found" HTML files are in a non-standard location Tell Claude Code which directory contains your HTML files
Contrast checks say "CSS file not found" Stylesheets couldn't be resolved from HTML link tags Make sure CSS files are accessible at the paths referenced in your HTML
