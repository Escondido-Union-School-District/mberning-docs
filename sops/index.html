Claude Code — Standard Operating Procedures

Author: Mark Berning, EUSD  |  Last Updated: 2026-04-06

Operational procedures for Claude Code automation workflows. Each SOP is self-contained and includes a replication guide for anyone who wants to set up something similar in their own environment.

How This All Works

This section explains how these tools get built, maintained, and documented — from start to finish.

Starting a new project

Every project begins with an idea. Sometimes you know exactly what you want to build. Other times you're not even sure it's possible — you just want to talk it through. Either way works.

If you're exploring, open a terminal and start a conversation with Claude. No folder, no setup — just talk. "Is it possible to crawl school websites for broken links? How would that work?" Go back and forth as long as you need. Not every conversation becomes a project.

When you're ready to build, create a folder in ~/projects/eusd/, open it in a terminal, and start Claude with "I want to start a new project that..." From that one sentence, Claude sets up everything: version control, a GitHub repository, a shortcut command so you can reopen the project anytime by typing two or three letters, and a user guide that explains how the tool works. You don't need to remember any of this — just say "new project" and it happens. The details are in SOP-001.

Working during the day

During the day, you work on projects by typing your shortcut alias (like pp for Parent Portal or lc for link checker) from any terminal. Claude writes code, runs it, fixes bugs, and commits everything to GitHub as you go. You don't need to think about saving, backing up, or version control — it's all handled automatically.

What happens at the end of each day

Three things run back to back every weekday afternoon. You don't need to do anything — just leave your machine on.

At 3:01 PM, a script on your PC looks at every project that had activity today and saves a snapshot of where things stand — what you were working on, what's next, any open questions. These snapshots are stored in Claude's memory so you can pick up right where you left off, even weeks later. See SOP-002.

At 3:30 PM, a cloud-based agent checks all your project repos on GitHub. If you made code changes that affect how a tool is used, it updates the user guide automatically. It also logs every commit to a "Recent Changes" section on each project's guide page, and publishes everything to a website where staff can browse all your tools. See SOP-003.

At 3:50 PM, another script on your PC pulls down all the changes the cloud agent made, then sends you a daily status email showing which projects had activity, what was updated, and which projects are being monitored.

Coming back to a project

When you return to a project — whether it's been a day or a month — type the alias and say "catch me up." Claude reads the saved snapshots and recent git history and gives you a briefing: where you left off, what's next, and any open questions. You don't need to read through code or git logs yourself.

Sharing with staff

Every project's user guide is published to a website that anyone in the district can access by clicking a link:

View all project guides

The guides explain how to use each tool step by step, no technical knowledge required. They're updated automatically every day and always reflect the latest version of the code.

SOP-001: New Project Setup

Exploring an idea Ready to build
  1. Open a terminal
  2. Run claude
  3. Talk through your idea
  4. If it has legs, move to Ready to build →
  1. Create a folder in ~/projects/eusd/ — the folder name matters because Claude derives the launch alias from its initials (e.g., link-checker → lc)
  2. cd into the folder
  3. Run claude
  4. Start your message with "I want to start a new project that..." — the phrase "new project" triggers a setup skill that handles git, aliases, GitHub repo, and user guide automatically.

What is it?

A checklist that runs automatically when you start a new project. You create a folder and describe what you want to build — Claude handles the rest: version control, a GitHub repository, keyboard shortcuts to launch the project, and a user guide if it's a work project. The full checklist is driven by a skill file so nothing gets missed.

What Problem Does This Solve?

Every new project needs the same handful of things before you start building. Some of these Claude would do on its own — like initializing git or creating a .gitignore. But others won't happen unless something tells Claude to do them: launch aliases (short commands like lc that open the project in one step) in both bash and PowerShell profiles, a GitHub repo, a user guide, and the ADA skill. Without this SOP, those get forgotten — you end up with a project that works but has no shortcut to open it, no repo backing it up, and no documentation.

How It Works

When you say "new project" in an empty directory (see step 4 in Ready to build above), Claude triggers the new-project skill. You don't need to remember a checklist — the skill handles it. Here's what it does:

  1. Asks about your idea if it needs more detail — you can go back and forth as long as you need before anything gets created
  2. Initializes git, creates CLAUDE.md and .gitignore
  3. Adds launch aliases to both bash and PowerShell profiles (e.g., lc to reopen the link checker project). Both profiles are needed because Claude Code runs in a bash shell, but you may also open terminals in PowerShell. See SOP-004 for details
  4. Creates a private GitHub repo and pushes the initial commit. Claude determines the GitHub owner from the folder path — projects in ~/projects/eusd/ go to the Escondido-Union-School-District org, projects elsewhere in ~/projects/ go to your personal GitHub account
  5. Creates a user guide at ~/projects/eusd/<project-name>/docs/user-guide.html for all work projects
  6. Installs the ADA accessibility skill if it's a web project
  7. Confirms everything it did and gives you the alias

You create the folder and say "new project" — Claude handles everything after that.

What to Expect

During setup: After you've finished discussing the idea, Claude runs through the setup checklist — creating files, setting up the repo, adding aliases. You don't need to do anything during this part.

When it's done, you'll have:

  • CLAUDE.md, .gitignore, and an initial commit in your project folder
  • A private GitHub repo with the code pushed (under the EUSD org for work projects, your personal account otherwise)
  • A launch alias — Claude picks the alias and tells you what it is when setup is complete. Open any terminal, type it from any directory, and it navigates to the project and starts Claude Code automatically (see SOP-004)
  • A user guide at ~/projects/eusd/<project-name>/docs/user-guide.html (all work projects)
  • The ADA build-time skill installed (if it's a web project)

How to verify: Open a new terminal and type the alias Claude gave you. You should land in the project folder with a Claude Code session running.

Gotchas

  • Alias conflicts: If the project initials collide with an existing alias or system command, Claude will double the last letter (e.g., fp → fpp) or ask you to pick one.
  • New terminal required: After aliases are added to your profiles, you need to open a new terminal (or source ~/.bashrc) for the alias to be available.

SOP-002: EOD Daily Summary System

What is it?

At the end of each workday, a PowerShell script runs automatically on your PC at 3:01 PM via Windows Task Scheduler. It does two things: first, it looks at every project that had activity and saves a snapshot of where things stand — what was being worked on, what's next, and any open questions. These snapshots are saved as session notes in Claude Code's memory system. Second, it generates a single daily summary that covers all projects in one file, so you can see everything at a glance.

You can step away from a project for days or weeks and pick up right where you left off by asking Claude "catch me up" in any project.

What Problem Does This Solve?

When you juggle multiple projects, you inevitably leave sessions mid-task. Coming back days later, git history and code alone don't recapture why you were doing something or where you were heading. The EOD system automatically captures that context so you don't have to remember to save it yourself.

How It Works

The script (~/projects/daily-summaries/eod-summary.ps1) runs in two phases, back to back:

Phase 1 — Per-project capture: The script loops through each project one at a time. For each project that has new commits since the last capture, it forks the most recent Claude session (open or closed — it reads conversation history from disk), commits and pushes any uncommitted changes (skipping .env and secrets), and appends timestamped session notes to ~/.claude/projects/<key>/memory/session-notes.md. Projects with no new commits are skipped entirely. Each project has a 180-second timeout — if a session hangs, it gets killed so it doesn't block the rest.

Phase 2 — Cross-project summary: Immediately after Phase 1 finishes, a separate Claude instance scans all projects and aggregates session notes, git activity, and file changes into a single daily markdown briefing at ~/projects/daily-summaries/YYYY-MM-DD.md.

There is no delay between phases — Phase 2 starts as soon as the last project in Phase 1 completes (or times out).

Note: This is separate from Claude Code's built-in auto memory, which saves technical learnings (build commands, patterns, debugging tips). The EOD system captures project progress — where you left off, next steps, open questions, key decisions — which auto memory does not track.

Retention: The two phases produce different outputs with different lifespans:

  • Session notes (Phase 1) — the per-project snapshots that record where you left off, next steps, and open questions. These are kept forever because you might not revisit a project for months, and the context needs to be there when you do. This is what Claude reads when you say "catch me up."
  • Daily summaries (Phase 2) — the single file that rolls up all projects into one overview for the day. These are deleted after 30 days because they're only useful for a quick glance at recent activity, not long-term reference.

How to Retrieve Notes

When you come back to a project (even weeks or months later), say "catch me up" or "where was I" in any project terminal. Claude reads the session notes at ~/.claude/projects/<key>/memory/session-notes.md and recent git history, then gives you a briefing of where things stand and what's next.

This goes beyond what Claude remembers on its own. Claude's built-in auto memory saves technical learnings — things like which build commands work or how a particular API behaves. But it doesn't track what you were doing, where you were heading, or what decisions you made along the way. Session notes fill that gap. Without them, Claude would start a new session knowing the codebase but having no idea what you were in the middle of or what you planned to do next.

What to Expect

Every weekday at 3:01 PM, the scheduled task runs silently in the background. Your open terminals are untouched — it forks sessions rather than interrupting them.

If your PC is asleep at 3:01 PM, the task will run when the PC wakes up (the scheduled task is configured with StartWhenAvailable). If the PC stays asleep or off for the rest of the day, that day's notes and summary will be skipped entirely.

What gets created:

  • Per-project session notes at ~/.claude/projects/<key>/memory/session-notes.md — append-only log of where you left off, what's next, open questions
  • Daily summary at ~/projects/daily-summaries/YYYY-MM-DD.md — one file per day covering all projects
  • Execution log at ~/projects/daily-summaries/eod-log.txt — shows Done/Failed/Skipped/Timeout per project per run

Session notes live in Claude Code's memory system (~/.claude/projects/). Daily summaries, the script, and the log all live in ~/projects/daily-summaries/ — a plain folder, not a git repo or Claude Code project.

To confirm the task ran: Check ~/projects/daily-summaries/eod-log.txt — it shows Done/Skipped/Failed/Timeout per project per run.

Gotchas

  • Say "eod" or "sv" before /clear. If you /clear a session and then close it, the automated 3:01 PM task will fork an empty conversation with no context to capture. Always save notes before clearing.
  • A project needs at least one Claude session before the automated fork can capture it. If you see "no session to continue" in ~/projects/daily-summaries/eod-log.txt, just open that project in Claude Code and do some work first.
  • Duplicate detection: If you say "eod" manually and the 3:01 PM task fires within 10 minutes, the script skips that project to avoid near-duplicate notes.
  • New projects are auto-registered. The new-project skill (SOP-001) automatically adds new projects to the EOD script. If you created a project without using the skill and it's not showing up in the daily summary, tell Claude: "Add [project name] to the EOD script and pull script" — Claude will add it to the $projects array in ~/projects/daily-summaries/eod-summary.ps1 and the $repos array in ~/projects/daily-summaries/pull-all-repos.ps1.
  • Timeouts: Each project has a 180-second timeout (360 seconds for the summary phase). If a session hangs, it's killed and the script moves on. Check ~/projects/daily-summaries/eod-log.txt for "Timeout:" entries.

Replication Guide
Note: The instructions below are from a specific setup (8 EUSD projects, Windows 11, Claude Code CLI). Adapt file paths, project lists, and timing to match your own environment.

Step 1: Add session notes instructions to global CLAUDE.md

Add this to your global ~/CLAUDE.md so you can trigger notes manually by saying "eod":

Text to add:

## Session Notes

When the user says "eod", "end of day", "save notes", "sv", or "done for the day":

1. If this is a git repo, check for uncommitted changes. If any, commit
   with a descriptive message and push to remote. Skip .env and secrets.
2. APPEND a timestamped entry to session-notes.md in this project's
   memory directory. Do not overwrite. Do not remove old entries.
3. Update the project's MEMORY.md index if needed.
4. Confirm: "Session notes saved. Safe to /clear or start a new session."

When the user says "catch me up" or "where was I", read session-notes.md
and summarize the most recent entries.

Step 2: Create the EOD script

The production script is at ~/projects/daily-summaries/eod-summary.ps1. Use it as a reference. Key architecture:

  • A $projects array listing all project paths
  • Phase 1 loops through projects, piping an "eod" prompt to claude -p -c --fork-session --dangerously-skip-permissions --model sonnet
  • Phase 2 uses a fresh claude -p with --add-dir to scan all projects and write a daily summary
  • Timeouts via System.Diagnostics.Process + WaitForExit + taskkill /F /T
  • Duplicate detection: skip if ~/.claude/projects/<key>/memory/session-notes.md modified in last 10 minutes
  • Append-only logging to ~/projects/daily-summaries/eod-log.txt

Step 3: Schedule the task

$action = New-ScheduledTaskAction `
    -Execute 'powershell.exe' `
    -Argument '-ExecutionPolicy Bypass -File "C:\Users\mberning\projects\daily-summaries\eod-summary.ps1"'

$trigger = New-ScheduledTaskTrigger `
    -Weekly -DaysOfWeek Monday,Tuesday,Wednesday,Thursday,Friday `
    -At '3:01PM'

Register-ScheduledTask -TaskName 'Claude EOD Summary' `
    -Action $action -Trigger $trigger

Step 4: Test the pipeline

  1. Open a project in Claude Code and do some work
  2. Say "eod" and verify ~/.claude/projects/<key>/memory/session-notes.md was created
  3. Run the script manually and check ~/projects/daily-summaries/ for today's file
  4. Check ~/projects/daily-summaries/eod-log.txt for errors
  5. Next day, say "catch me up" and verify it works

SOP-003: Project User Guides

View all project guides

What is it?

Every work project gets a user guide — a printable HTML page that explains how to use the tool, written for non-technical staff. The user guide is created automatically when the project is set up, and a cloud-scheduled agent keeps it up to date every weekday at 3:30 PM — updating instructions when the code changes, logging all commits to a "Recent Changes" section, and publishing everything to a shared website where staff can browse all project guides.

What Problem Does This Solve?

Without documentation, only the person who built a tool knows how to use it. If someone else needs to run the tool — or you come back to it months later — there's nothing to reference. This SOP ensures every project with a user-facing tool has a printable user guide for end users.

How It Works

Each project gets a docs/user-guide.html — a self-contained, printable HTML guide that lets someone use the tool without Claude Code. Written for a non-technical audience.

All user guides are built from a shared template enforced by the eusd-user-guide Claude Code skill. Every guide looks identical — same CSS, same section order, same component patterns. Only the content changes.

How It's Triggered

  • Updates guide content if code changes affect user-facing behavior
  • Adds all commits to the "Project recent changes" section on each guide (always keeps the 10 most recent entries plus all entries from the last 30 days)
  • Appends to the permanent docs/changelog.md in each project (full history, never pruned)
  • Syncs updated guides to the mberning-docs repo for GitHub Pages hosting

What to Expect

User guides are self-contained HTML pages with a consistent structure and styling across all projects. Each includes a "Print this guide" link and, on GitHub Pages, an "All tools" link back to the index page.

When a guide is created or updated, it's committed and pushed in the same session. Claude won't leave it as an untracked file.

Every guide includes a collapsible "Project recent changes" section showing the 10 most recent commits (always kept regardless of age) plus all commits from the last 30 days. A permanent changelog is also maintained at docs/changelog.md in each project (full history, never pruned). Guides are also published to GitHub Pages at escondido-union-school-district.github.io/mberning-docs for easy sharing with internal staff.

Gotchas

  • You don't need to update guides yourself. The 3:30 PM cloud agent is the safety net — it catches any code changes that affect the guide and updates it automatically. If you want to update a guide during a session you can, but you don't have to. Guides may be up to 24 hours behind code changes until the next automated run.
  • Cloud trigger needs GitHub write access. The 3:30 PM agent runs in Anthropic's cloud and pushes changes to your repos. If it stops working, check that the GitHub Integration at GitHub > Org Settings > Installations still has write access to your repos.

Step 2: Create the Claude Code Skill

The skill ensures every guide Claude generates uses the same CSS and structure. Create two files:

  • ~/.claude/skills/eusd-user-guide/SKILL.md — rules, required section order, component patterns, common mistakes table. The description field should trigger on creating or updating docs/user-guide.html.
  • ~/.claude/skills/eusd-user-guide/template.html — complete HTML/CSS skeleton. Claude copies the <style> block verbatim and fills in project-specific content.

Pick one existing guide as the "gold standard" (we use PDF Form Prep) and extract its CSS and structure into the template. The skill's SKILL.md should list:

  • The exact section order (What It Does, Prerequisites, Step-by-Step, CLI Reference, Folder Structure, Understanding Output, Troubleshooting, Quick Reference Card)
  • HTML patterns for each component (step-num, callout, callout-info, quick reference card)
  • A "Common Mistakes" table listing things Claude tends to get wrong (dark code blocks, wrong accent color, adding TOCs, etc.)
How it works: Claude Code discovers skills by matching their description field to the current task. When you ask it to create a user guide, it automatically loads the skill, reads the template, and follows the rules. No manual invocation needed.

Step 3: Generate a Guide for an Existing Project

Recommended prompt:

Create a printable user guide at docs/user-guide.html for this project.
Read the code to understand what the tool does, its CLI flags/options,
prerequisites, folder structure, and common errors. The guide should be
self-contained so someone can use the tool without Claude Code running.
Include a "Print this guide" link, step-by-step usage instructions,
and a troubleshooting section.

Claude Code will invoke the eusd-user-guide skill automatically and produce a guide matching the template. If an existing guide uses a different style, use:

Prompt to standardize an existing guide:

Rewrite docs/user-guide.html to match the EUSD standard template.
Keep all content but replace the CSS and structure.

SOP-004: Project Launch Aliases

What is it?

A shortcut system that lets you open any project from any terminal by typing two or three letters. For example, type pp and you're immediately in the Parent Portal project with Claude Code running. No need to remember folder paths or type long commands. Aliases are created automatically when you start a new project.

What Problem Does This Solve?

Without aliases, launching a project means remembering (or looking up) the folder path, cd-ing into it, and typing claude --name 'Project Name'. The --name flag matters because it gives sessions readable names in /resume and Claude Code Remote — without it, sessions show as opaque IDs and you have to guess which is which. Aliases eliminate all of this: type two or three letters from any terminal and you're in.

How It Works

Each project gets a short alias (typically its initials) defined in both shells:

ShellFileExample
Bash ~/.bashrc alias pp="cd ~/projects/eusd/parent-portal && claude --name 'Parent Portal'"
PowerShell ~/Documents/WindowsPowerShell/Microsoft.PowerShell_profile.ps1 function pp { Set-Location ~/projects/eusd/parent-portal; claude --name 'Parent Portal' }

Typing pp from any directory changes into the project folder and starts Claude Code with a named session.

How It's Triggered

Aliases are added automatically when a new project is created via SOP-001. The new-project skill derives the alias from the project name initials, checks for conflicts, adds it to both profile files, and tells you what it picked.

You can also ask Claude to add an alias manually at any time — the global ~/CLAUDE.md has the naming rules and conflict-check process.

What to Expect

After an alias is added, open a new terminal (or source ~/.bashrc) and type the alias. You'll land in the project folder with a Claude Code session running under the project's name.

Gotchas

  • New terminal required. Aliases aren't available in terminals that were already open when the alias was added.
  • Naming conflicts. If the initials collide with an existing alias or system command, Claude doubles the last letter (e.g., fp → fpp) or asks you to pick one. Minimum 2 characters.
  • Both files must stay in sync. If you edit one profile manually, update the other too. The authoritative source is always ~/.bashrc and the PowerShell profile.

SOP-005: SOP Accuracy & Effectiveness Audits

Currently disabled. Both the local scheduled task and the cloud triggers for this SOP are turned off. The content below describes how it worked when active. It needs to be reviewed and re-enabled when ready.

What Problem Does This Solve?

SOPs describe files, paths, processes, and configurations that change over time. Without regular verification, they drift from reality — file paths get renamed, components get reorganized, processes evolve while the documentation stays frozen. This SOP automatically checks itself (and all the other SOPs) for accuracy, and periodically asks whether each SOP is still worth having.

How It Works

A single PowerShell script runs two audits:

  1. Weekly accuracy audit (every Monday): Reads this SOPs file, extracts every verifiable claim (file paths, component names, process descriptions), and cross-references them against the actual files on this machine. Categorizes each claim as Verified, Inaccurate, Stale, or Unverifiable.
  2. Monthly effectiveness review (1st of each month): Evaluates whether each SOP is solving a real problem, whether the complexity is proportional to the risk, and whether any processes are redundant or over-engineered. Each SOP gets a verdict: Keep, Simplify, Merge, or Eliminate.

Both audits write a report, commit and push it, and open a GitHub issue so you get an email notification.

How It's Triggered

Automatic: Windows Task Scheduler runs ~/projects/sops/sop-audit.ps1 every Monday at 9:00 AM. The monthly review only fires when Monday falls on the 1st.

If your PC is asleep at 9 AM, the task runs when the PC wakes up (StartWhenAvailable). If the PC stays off all day Monday, that week's audit is skipped.

To run manually:

powershell -ExecutionPolicy Bypass -File ~/projects/sops/sop-audit.ps1

What to Expect

Every Monday, you'll get a GitHub notification email with a subject like:

SOP Audit 2026-03-31: 3 issues found (2 inaccurate, 1 stale)
SOP Audit 2026-04-07: All clear

On the 1st of each month (if it's a Monday), you'll also get:

SOP Effectiveness Review 2026-04-01: 2 SOPs recommended for simplification

For the full report, pull the SOPs repo and read ~/projects/sops/sop-audit-report.md or ~/projects/sops/sop-effectiveness-review.md. Reports are overwritten each run. The execution log at ~/projects/sops/sop-audit-log.txt is append-only.

If a finding requires an SOP update, edit ~/projects/sops/claude-code-sops.html, commit, and push. The next week's audit will verify the fix.

Gotchas

  • Task Scheduler configuration can't be verified. The audit runs locally with access to all project files, profiles, and scripts — but it can't query Task Scheduler from bash. Scheduled task claims are marked "Unverifiable."
  • New SOPs need --add-dir. If a new SOP references files in a directory the audit doesn't currently check, add an --add-dir flag to the relevant claude -p call in ~/projects/sops/sop-audit.ps1.
  • The audit doesn't fix anything. It reports issues and opens a GitHub issue. You still need to edit the SOPs file manually.
