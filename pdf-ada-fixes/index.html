What This Tool Does

This tool automatically fixes six common PDF accessibility issues required for ADA/Section 508 compliance, plus two additional fixes available via the optional --vision-fix flag:

#FixWhat It Does
1Figure ImagesUses Google Gemini AI to classify each image as decorative or content, shows you the findings, and lets you correct any before applying. Content images get descriptive alt text; decorative images are artifacted so screen readers skip them. Controlled by --images (see below).
2Document LanguageSets the document language (default en-US; pass --lang es-ES for Spanish or any other BCP-47 tag)
3Tab OrderSets structure-based tab order (/Tabs /S) on every page so keyboard/screen-reader tab navigation follows the reading order (Acrobat Pro: Tab Order)
4Document TitleExtracts the first <H1> heading and sets it as the document title
5Table HeadersMarks table header cells so screen readers can navigate tables
6Underscore Fill-in LinesWraps runs of 3 or more underscore characters in PDF artifact blocks so screen readers skip them. The visible line stays on the page for sighted users to write on.
6Table Regularity (opt-in: --vision-fix)Uses Claude vision to detect rows where the cell count doesn't match the table's actual visual width and writes the correct ColSpan/RowSpan (Acrobat Pro: Tables > Regularity)
7Heading Hierarchy (opt-in: --vision-fix)Uses Claude vision to demote headings that aren't really headings (e.g., a closing courtesy phrase Word stamped as <H3>) so the heading outline matches the document's actual structure (Acrobat Pro: Appropriate Nesting)
Requirement: Input PDFs must be tagged PDFs (exported from Word with "Create bookmarks using Headings" or similar). Untagged/scanned PDFs will not work.

The --images Option

This controls how the tool treats the images (<Figure> tags) in a PDF. Not every image needs a description — purely decorative graphics (background shapes, borders, color blocks, dividers) should be artifacted so screen readers skip them entirely. Giving a decorative image alt text clutters the reading order and gets flagged by accessibility checkers like PAC.

OptionWhat It Does
--images classify
(default)		 The AI looks at each image and decides DECORATIVE or CONTENT. It prints a numbered list of its findings, then — when you're at a terminal — asks you to confirm. Press Enter to accept all, or type the figure numbers you want to flip (for example 2 5 6) and follow the prompts. Content images get alt text; decorative images are artifacted.
--images decorative Treats every image as decorative and artifacts them all. No AI calls, no prompts. Use this for flyers and posters whose images are all background art.
--images alt The original behavior: generate alt text for every image, no classification. Use only if you know every image is meaningful.
Reviewing the AI's guesses: The classifier can mistake a decorative shape for content when text sits on top of it. That's what the confirm step is for — glance at the list and flip anything wrong before it's applied. If you flip a decorative image to content, you'll be asked to type its alt text.

The --headers Option

This controls how table headers are marked. Choose based on your table layout:

OptionUse WhenWhat It Does
--headers row First row contains column headings (most common) Marks first row cells as <TH Scope=Column>
--headers column First column contains row headings Marks first cell in each row as <TH Scope=Row>
--headers both Both first row and first column are headings Applies both row and column header fixes
--headers none Tables don't have headers or are already correct Skips table header fixing entirely
Default: If you don't specify --headers, it defaults to row (first row = column headers). This is correct for most EUSD implementation plan templates.

The --vision-fix Option

The five default steps fix the most common ADA issues using rule-based logic alone. But two failures from Acrobat Pro's Accessibility Checker need a model that can actually look at the page:

  • Tables > Regularity — fires when a row's tagged cell count doesn't match the table's visual column count. Fix requires deciding which cell should span multiple columns (or rows), which depends on what the page actually looks like.
  • Appropriate Nesting — fires when a heading is in the wrong place (e.g., Word silently stamped <H3> on a closing courtesy phrase). Fix requires reading the surrounding context to decide what the text actually is.

Pass --vision-fix to enable a Claude vision pass that handles both:

python remediate.py input/ --vision-fix

Prerequisites

  • ANTHROPIC_API_KEY in .env (paid; budget a few cents per PDF)
  • Internet access (calls the Anthropic API for each detected issue)

What you get

  • Steps 6 and 7 run after the existing 5 steps. Each detected issue (irregular row, suspect heading) gets its own Claude call with a rendered PNG of the relevant page plus a structured snippet of the tag tree.
  • Sidecar audit log at output/<basename>_remediated.fixes.json. Every applied or skipped change is recorded with Claude's reasoning, so a reviewer can spot-check decisions in seconds without re-running the model.
Opt-in by design. Don't burn tokens on PDFs that already pass Acrobat's accessibility check. Run the tool without --vision-fix first; only re-run with the flag if Acrobat Pro flags Tables > Regularity or Appropriate Nesting failures.

The --review-tables Option

Word frequently emits "matrix-style" tables (e.g., a survey grid where rows are questions and columns are response choices) as <Table> elements with an empty top-left <TH> corner cell. Screen readers handle these poorly because each data cell announces its row header but the column header above it is empty. The accessibility fix is to restructure the table into a list (<L> / <LI>) where each row becomes a list item and each cell carries an /ActualText entry that injects the column header into the cell's announced text.

Because that restructure is destructive (the visual table stays, but the tag tree is rewritten), it runs in a two-phase workflow: queue, then walk and apply. You see Claude's proposal for every candidate before anything mutates the PDF.

Phase 1 — Queue candidates

python remediate.py "input/YourFile.pdf" --review-tables

This runs the existing 5 steps, then a Step 8 pass that:

  • Detects every <Table> with empty <TH> corner cells.
  • Asks Claude to look at a rendered page image and propose a Table→List restructure for each one (including the new column header per cell).
  • Writes the queue to output/<basename>_remediated.review.json.
  • Saves rendered page PNGs (one per candidate page) to output/<basename>_remediated.review/.

The remediated PDF is still saved as output/<basename>_remediated.pdf, but the queued restructures are not applied yet.

Phase 2 — Walk the queue and apply

python remediate.py review "output/YourFile_remediated.review.json"

The tool walks each proposal interactively. For each entry it shows the page image path, the table snippet, and Claude's proposed list structure, then prompts:

KeyAction
aAccept — queue this restructure to apply
sSkip — leave the table alone
nNote and skip — record a reason in the queue, leave table alone
qQuit — stops the walk; pending entries remain so you can resume

Each decision is persisted to the queue file immediately, so Ctrl-C or q is safe. Re-run the same review command to resume where you left off. When every entry has a decision, accepted restructures are applied to the remediated PDF in place and recorded in the audit log.

Audit log

Restructure entries are appended to the same output/<basename>_remediated.fixes.json audit log used by --vision-fix, so a reviewer can see every accepted change with Claude's reasoning and the original/new tag structure.

Prerequisites

  • ANTHROPIC_API_KEY in .env (paid; budget a few cents per candidate)
  • Internet access (calls the Anthropic API for each candidate during the queue pass)

Auto-Tagging Untagged PDFs (autotag.py)

remediate.py only fixes tags that already exist — it refuses a PDF that has no tag structure at all. Graphically-designed flyers (from Canva, InDesign, etc.) often arrive untagged: they have selectable text and decorative images but no structure tree, and Acrobat's "Autotag" does a poor job on them. autotag.py builds the tags from scratch.

For each page it renders the page, then asks Claude (vision) to look at the flyer and return the correct reading order, a role for each piece of text (Title, Heading, Body), and the correctly transcribed text — reading the words off the image, so it fixes garbled accents/punctuation that a broken text layer would otherwise carry into the tags. It then shows you the detected outline to confirm before writing.

How it keeps the look identical: the original page is preserved as a background "artifact" (so the flyer renders exactly as before and its decorative images are skipped by screen readers), and an invisible tagged text layer carries the reading order. Screen-reader text comes from the Unicode the vision model read, so Spanish accents (ñ, á, ¡, ¿) are exact.

1 Run it on an untagged flyer

# English
python autotag.py "input/YourFlyer.pdf"

# Spanish (or other language)
python autotag.py "input/YourFlyer_Spanish.pdf" --lang es-ES

2 Confirm the detected outline

The tool prints the reading order with a role per line, then prompts:

KeyAction
EnterAccept — build the tagged PDF as shown
eEdit an item — change a line's role (H1/H2/H3/P) or drop it
tSet title — type a different document title
qAbort — write nothing, leave the file in input/

The result is saved as output/<basename>_tagged.pdf; the original is moved to processed/.

FlagDefaultDescription
--langen-USDocument language (BCP-47); use es-ES for Spanish
--title(vision-chosen)Override the document title instead of using the one the model composes
--yesoffSkip the confirm prompt and apply the detected structure as-is
Requirements: autotag.py needs ANTHROPIC_API_KEY in .env (paid; budget a few cents per page) and a PDF that has a real text layer. It is only for untagged PDFs — if a PDF is already tagged, use remediate.py instead.

Processing Multiple Files

The easiest way to process all PDFs at once is to pass the folder path:

python remediate.py input/

This processes every .pdf file in the folder and prints a batch summary when finished. (autotag.py accepts a folder the same way.)
